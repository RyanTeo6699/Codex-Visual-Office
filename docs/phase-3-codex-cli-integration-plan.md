# Phase 3 - Codex CLI Integration Plan

## Proposed Architecture

Phase 3 should add a narrow local Codex CLI integration layer on top of the Phase 2 local task system.

Recommended layers:

- UI pages/components remain React and Next.js.
- Server-only local integration code owns Codex CLI detection and later dispatch.
- SQLite stores safe settings, dispatch records, task status changes, and task events.
- Existing `agent_seats`, `tasks`, and `task_events` remain the main visual state surface.
- No OpenAI API or cloud service is introduced.

The integration should progress from read-only detection to dry-run handoff before any real CLI execution is added.

## Step 1 CLI Availability Detection

Step 1 should detect whether a `codex` command is available without running tasks.

Recommended checks:

- Resolve a configured CLI path if one exists.
- Otherwise detect whether `codex` is available on the server process PATH.
- Read version with a safe version-only invocation if supported.
- Return a status object such as:

```ts
type CodexCliAvailability =
  | { status: "installed"; pathLabel: string; version?: string; authStatus: "available" | "unknown" }
  | { status: "not_found"; authStatus: "unknown" }
  | { status: "error"; message: string; authStatus: "unknown" };
```

Step 1 must not execute Codex tasks, run arbitrary commands, or add a reusable runner.

## Avoiding Secret Reads

The app must not read:

- `~/.codex/auth.json`
- OpenAI API keys
- OAuth tokens
- CLI auth token files
- credential environment variables

The app should infer safe auth status only from non-secret CLI behavior or user-provided status confirmation.

If the CLI cannot expose auth status safely, the UI should show `unknown` and tell the user that the app does not inspect credentials.

## Safe Auth Status Representation

Auth status should be represented as:

- `available`
- `not_available`
- `unknown`

The app should never display raw credential paths, token values, token prefixes, account identifiers, or API keys.

## CLI Path and Settings Storage

Safe SQLite settings may include:

- configured CLI path label or absolute path selected by user
- preferred local project path
- last successful CLI availability check timestamp
- last seen CLI version string
- auth status label

Settings must not include:

- tokens
- passwords
- API keys
- OAuth values
- raw auth file contents

## Mapping Runtime Status to Existing Tables

Phase 3 should map Codex runtime state onto the existing local data model:

### `agent_seats`

- `idle`: no assigned Codex work.
- `planning`: prompt prepared or dry-run handoff ready.
- `editing`: Codex task is actively running.
- `running_checks`: Codex appears to be validating output.
- `waiting_review`: Codex task completed and needs user review.
- `blocked`: dispatch failed or user intervention is needed.
- `done`: task completed and accepted.

### `tasks`

- `ready`: prompt handoff prepared.
- `running`: Codex CLI task running.
- `waiting_review`: Codex completed and needs review.
- `blocked`: dispatch failed or user rejected output.
- `done`: review accepted.

### `task_events`

Task events should record:

- prompt prepared
- dry-run handoff saved
- dispatch confirmed
- Codex started
- output milestone received
- Codex completed
- Codex failed
- review decision made

## Visual Changes to Codex Work Pods

Phase 3 should visually extend existing work pods, not redesign the app.

Recommended changes:

- Add a small CLI status indicator on Codex seats.
- Show installed/not found state in a restrained badge.
- Show runtime state in existing seat status language.
- Show task progress through existing task progress and event feed patterns.
- Avoid new dashboard-style surfaces unless a narrow control is required.

## Later Task Dispatch Flow

Recommended dispatch flow:

1. User opens a task or review room.
2. App generates a Codex prompt preview from local task data.
3. User reviews prompt and target project path.
4. User confirms dry-run handoff or real dispatch, depending on phase step.
5. Scoped runner invokes only `codex`.
6. Output is streamed into UI.
7. Task events are persisted.
8. Task status moves to waiting review or blocked.

## SQLite Persistence

Phase 3 may persist:

- CLI availability status.
- safe version string.
- safe auth status label.
- configured CLI path.
- approved project path.
- prompt preview/dry-run dispatch records.
- dispatch status.
- task events generated by Codex runtime milestones.
- task status and agent seat status transitions.

## Data That Must Not Be Persisted

Phase 3 must not persist:

- OpenAI API keys.
- Codex auth tokens.
- `~/.codex/auth.json` contents.
- OAuth tokens.
- passwords.
- arbitrary terminal output without redaction.
- full local project file contents.
- git diffs from real repositories.
- deployment credentials.

## What Remains Forbidden

- OpenAI API integration.
- GitHub API/App integration.
- Vercel integration.
- Supabase integration.
- Cloud sync.
- Auth/payment/billing.
- MCP server.
- `node-pty`.
- generic terminal runner.
- arbitrary shell commands.
- Git watcher.
- file watcher.
- automatic git push.
- automatic deploy.

## Risk Mitigations

### Credential Leakage

- Never read auth files.
- Never display token-like strings.
- Redact secret-looking output before persistence.
- Keep auth state coarse-grained.

### Arbitrary Command Execution

- Do not expose a free-form command field.
- Use a fixed Codex command invocation builder.
- Validate target paths against user-approved project paths.

### Accidental Repository Mutation

- Start with dry-run prompt handoff.
- Require explicit user confirmation before real dispatch.
- Do not run git commands or deploy commands.

### UI Scope Creep

- Extend existing work pods and review/project rooms.
- Avoid broad dashboards.
- Preserve the current visual office structure.

### Persistence Drift

- Keep Codex runtime state mapped to existing `tasks`, `agent_seats`, and `task_events`.
- Add new tables only if dispatch records cannot be represented cleanly through settings/events.

## Phase 3 Step Plan

### Step 1 - CLI Availability / Version / Auth Status Display Only

- Detect whether `codex` command exists.
- Read version if safe.
- Display installed/not found.
- Do not run Codex tasks.
- Do not execute arbitrary commands.
- Do not read auth token contents.
- Do not add runner yet.

### Step 2 - Task Prompt Handoff / Dry-Run

- Generate and preview Codex prompt from task data.
- Mark task as ready for Codex.
- Copy prompt or persist a dry-run dispatch record.
- Do not execute Codex yet.

### Step 3 - Scoped CLI Runner

- Add explicit user-confirmed dispatch.
- Allow only `codex` command execution.
- Run only in user-approved project path.
- Save task event logs.
- No arbitrary shell command execution.
- No auto git push.
- No auto deploy.

### Step 4 - Output Stream + Task Event Logging

- Capture output safely.
- Display Codex work status.
- Persist events.
- Do not expose secrets.

### Step 5 - Closeout

- Verify boundaries.
- Document known limitations.
- Confirm no cloud integrations.
