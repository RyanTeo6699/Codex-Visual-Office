# Phase 10 - Real Project Workspace Hardening Scope Lock

## Phase Name

Phase 10 - Real Project Workspace Hardening

## Goal

Strengthen local-first real project workspace management so Codex Visual Office can represent approved local project workspaces with clearer safety, health, and navigation boundaries.

Phase 10 is a workspace hardening phase. It improves how already approved project workspace paths are represented, explained, and guarded in the UI. It does not introduce automatic discovery, source inspection, command execution, cloud services, team systems, or Codex automation.

## Core Positioning

Phase 10 moves the product closer to real local workspace readiness while preserving user approval and bounded observation.

The app may know that a project has an approved path and may show bounded metadata already stored in the local app model. It must not inspect the workspace source tree, discover projects on its own, or infer project details by reading files from approved paths.

## Allowed

- Local-first workspace management UX hardening.
- Approved project path state presentation.
- Clearer project workspace safety boundaries.
- Project health summary model using bounded local app state.
- Office Home project workspace health summaries.
- Project Room workspace boundary and health summaries.
- Settings approved path UX improvements.
- Status labels for approved path, missing path, unsafe path, or unconfigured path states.
- Mock or derived UI state from existing local app records.
- Documentation and roadmap status updates.
- Static verification planning and documentation.

- Reading `package.json` from approved paths.

- New Quality Gate commands.

- OpenAI API.

## Workspace Boundary Rules

Approved paths are user-declared boundaries, not permission for broad inspection.

Phase 10 may show:

- Whether a project has an approved path recorded.
- Whether the app model considers that path active, missing, unconfigured, or blocked.
- Whether local workspace actions are unavailable because approval is missing.
- Bounded project health summary fields stored or computed from app-owned records.
- User-facing explanations of what the app will and will not inspect.

Phase 10 must not show:

- Files discovered by scanning the approved path.
- Dependency or framework details read from `package.json` or config files.
- Git branch, status, diff, or commit information generated by new automatic Git commands.
- Quality results generated by new automatic command execution.
- Any token, env, auth, credential, or secret-derived state.

## No-Scan Policy

The app must not scan the user's disk.

This includes:

- No full disk scan.
- No home directory scan.
- No workspace directory crawl.
- No automatic project discovery from common folders.
- No background watcher used to discover projects.
- No automatic indexing of source files, docs, configs, lockfiles, or manifests.

Users must explicitly add or approve project paths through the existing local-first approval UX.

## No-Source-Read Policy

Approved paths are not approval to read source files.

Phase 10 must not read from approved paths to determine project details. In particular, it must not read:

- `package.json`
- Source files.
- Lockfiles.
- Framework configs.
- Build configs.
- `.git` internals.
- `.env` or `.env.local`.
- Auth, token, or credential files.

Project metadata shown in Phase 10 must come from app-owned local records, mock records, manual labels, or bounded status already captured by earlier approved flows.

## No-Auto-Execution Policy

Phase 10 must not automatically execute any workflow.

Forbidden automatic execution includes:

- Codex CLI or Codex App tasks.
- Git commands.
- Quality Gate commands.
- Build commands.
- Test commands.
- Terminal commands.
- Shell scripts.
- Package manager scripts.
- Deployment commands.

Any future execution expansion requires a separate GM-approved phase and a dedicated safety scope lock.

## Data Boundary

Phase 10 may use the local app's own stored project records and status records.

It must not expand the data boundary into:

- Source tree reads.
- Secret reads.
- Environment reads.
- Cloud sync.
- External APIs.
- External account systems.
- Team or permission systems.
- Remote project import.

## Acceptance Criteria

Phase 10 passes only if:

- Office Home clearly distinguishes configured, unconfigured, healthy, blocked, and review-relevant project workspaces using bounded local app state.
- Project Room clearly explains approved path state and workspace health without scanning or reading project files.
- Settings approved path UX makes the no-scan and no-source-read boundary visible.
- Project health summaries use app-owned local records, not workspace source inspection.
- No forbidden integrations or execution paths are added.
- No auth, env, token, or credential files are read.
- No Tauri production, Electron, installer, signing, or auto updater work is added.
- Phase 11 is not started.

## Failure Criteria

Phase 10 fails if it:

- Scans the disk or auto-discovers projects.
- Reads `package.json` or source files from approved paths.
- Reads env, auth, token, credential, or secret files.
- Adds automatic Codex, Git, Quality Gate, build, test, shell, terminal, or deployment execution.
- Adds cloud, GitHub, Vercel, Supabase, Firebase, auth, payment, billing, team workspace, MCP, ChatGPT App, or OpenAI API features.
- Adds Tauri production packaging, Electron, installer, signing, or auto updater behavior.
- Starts Phase 11.
